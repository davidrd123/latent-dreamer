## The shift

The center of gravity is moving.

The project is no longer being held primarily as:
- a conducted performance instrument
- a Mueller-faithful reconstruction
- a benchmark program whose main question is "what branch next?"

It is increasingly being held as:
- a way of working with LLMs that have persistent inner state
- a collaborative companion / daemon / thinking partner
- a substrate for memories, concerns, reactions, and unresolved
  questions that continue between sessions

That does not mean the current kernel work was wrong.
It means the reason for doing it is changing.

---

## What remains valuable from the current build

The current Mueller-shaped kernel is still valuable.

Not because it must remain the final architecture, but because it has
already provided:

- concern competition
- episodic accumulation that changes later behavior
- typed cognitive strategies
- memory hygiene / membrane discipline
- a real place where research ideas can land

Those are likely still load-bearing for the collaborative case.

The current system is therefore:
- a substrate
- a test bed
- a place to see what distinctions actually earn their keep

not necessarily the final form of the companion.

---

## The new question

The main question is shifting from:

"How do we finish recapitulating Mueller honestly?"

to:

"Which parts of this architecture are the real discipline required for
persistent collaborative inner life, and which parts are historical
baggage from Mueller or from the performance framing?"

That means questioning:
- the family taxonomy
- coincidence-mark retrieval as the main retrieval story
- the current rule-graph role
- the hardness of the kernel / LLM boundary
- whether "family plan" is still the right central unit for
  collaborative work

without throwing away the discipline that made the current build real.

---

## What this means practically

### 1. Understand the current machine end to end

Before changing direction, understand the current code and dynamics in
motion.

This is not only about tuning parameters. It is about being able to ask:
- does this distinction do real work?
- does this field matter?
- is this category earning its keep?
- is this mechanism actually better than a stronger LLM-facing
  alternative?

That is the role of a debug canopy / WorldDelta-style view.

### 2. Keep the current branch moving, but treat it as substrate work

The current Graffito / memory-ecology branch still matters because it is
making the substrate honest.

But it should now be interpreted as:
- proving persistence and accumulation
- proving meaningful typed dynamics
- creating a concrete thing that can be questioned

not as proof that the current specific architecture must be kept whole.

### 3. Re-open the architectural question at a higher level

The next round of questioning should be divergent, not only local and
mechanistic.

Key themes:
- what persistent inner state should feel like from the human side
- what the LLM actually needs from the kernel to be a better
  collaborator
- which Mueller mechanisms are load-bearing
- where LLM-native flexibility should lead more directly
- what mechanisms are missing for collaborative inner life

---

## The likely medium-term direction

The likely direction is not:
- tear the system down
- abandon the kernel
- stop caring about mechanistic integrity

It is more like:
- keep the structural discipline
- question the historical specifics
- make the interface to the LLM more central
- move toward a system that is personally useful as a collaborator

In other words:

**prosthetic inner life for collaborative LLMs**

is becoming the stronger framing than:

**conducted daydreaming instrument**

even if the latter remains one application and one source of discipline.

---

## What should remain live

Three tracks should stay live at once:

1. **Code understanding**
   - learn the current system deeply enough to steer it

2. **Mechanistic build**
   - keep the current branch honest enough that the substrate is real

3. **Divergent architectural questioning**
   - ask what the system should actually become if the collaborative
     companion use case is primary

The mistake would be collapsing to only one of these.

---

## Short version

The emerging pivot is:

Build enough of the current Mueller-shaped kernel to understand what is
truly load-bearing, then bend the project toward a personally useful,
persistent collaborative LLM with inner life, rather than assuming the
current form is the final destination.
