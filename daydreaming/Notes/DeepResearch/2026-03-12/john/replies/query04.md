John’s “temporal awareness is the Holy Grail” line is half right.

Temporal graph infrastructure is real. Temporal learner modeling for a personal knowledge vault mostly is not. The systems you asked about do four different jobs: Graphiti/Zep versions facts on graph edges; TigerGraph gives you high-throughput, time-stamped graph modeling; Neo4j gives you temporal property types plus manual modeling patterns; TerminusDB gives you immutable revision history and commit-level time travel. None of them, by itself, gives you a working model of “this person now understands recursion better than they did three weeks ago.” ([GitHub][1])

**Graphiti / Zep:** this is the closest thing on your list to a real temporal fact graph. Zep’s docs say facts live on edges and carry four timestamps: `created_at`, `valid_at`, `invalid_at`, and `expired_at`. When new episodes contradict old ones, Zep tries to invalidate the earlier fact and add new facts. Search supports datetime filtering over those edge timestamps, and recent releases added chronological batch ingestion so large episode sets preserve temporal order. Graphiti is also not just a paper toy here: the open-source engine is the core of Zep’s production context infrastructure. ([Zep Documentation][2])

The catch is that the temporal layer looks stronger on **edges** than on full entity history, and the OSS surface is still sharp-edged. An open January 2026 issue reports that node attributes are destructively upserted rather than historically versioned. Other open issues report datetime mismatches with Kuzu, broken `valid_at` search-filter query generation, a Neo4j backend regression after an update, and MCP-server runs that yielded blank disconnected nodes. There is also active demand for a “bring your own extraction pipeline” mode so Graphiti can be used mainly as a temporal graph engine instead of forcing LLM-heavy extraction end to end. So: real temporal capability, yes; boringly mature infrastructure, no. ([GitHub][3])

**TigerGraph:** real, deployed, and strong if your problem is large event streams and time-sensitive traversals. TigerGraph’s own examples show temporal hierarchies modeled with timestamped relationships and multi-edge discriminators so you can represent successive states between the same node pair. Their AML material describes sub-millisecond traversal on multi-hop time-sensitive paths with roughly 50 million events ingested per day. There is also `TemporalPyGTransform` for creating sequences of temporal subgraphs for ML, and GSQL has standard datetime functions. But this is not Graphiti-style semantic invalidation. You model time yourself with schema choices, timestamps, discriminators, and queries. Good database. Not a ready-made learner-memory system. ([TigerGraph][4])

**Neo4j:** deployed and useful, but you own the temporal semantics. Neo4j has built-in temporal value types in Cypher, official modeling guidance for time-based versioning with `validFrom`/`validTo`, and the old “timeline tree” pattern for traversing events by year/month/day. Bloom can filter and style nodes or relationships with temporal properties, and APOC is officially supported for temporal formatting/conversion utilities. What Neo4j does **not** give you out of the box is Graphiti-style automatic fact invalidation or a first-class temporal KG layer. Also, the old GraphAware TimeTree plugin is retired, and GraphAware itself discontinued the old framework/modules in 2021. So if you pick Neo4j, you are signing up to design the time model yourself. ([Graph Database & Analytics][5])

**TerminusDB:** real and usable, but it is doing a different kind of time. TerminusDB is Git-for-data: immutable history, branches, merges, diffs, commit graphs, and time travel to a chosen commit. Its docs explicitly frame the system around revision control and layers that record additions and deletions; current docs and late-2025 release notes show the project is active and still pushing versioned JSON/KG workflows. That makes it good for “what did the knowledge base look like at commit X?” It does **not** natively mean “when did this fact become true in the world?” unless you model that yourself. In other words: database-history time, not semantic-validity time. ([TerminusDB][6])

On the **personal-knowledge / learning** side, the picture is much thinner. I did **not** find a deployed local-first vault that infers a person’s evolving understanding of a concept by watching changes in their knowledge graph topology. What I found instead were partial approximations. Recall combines a knowledge graph with question generation, spaced repetition, knowledge stages, and time filters in graph view; RemNote combines a linked knowledge base with a graph view and per-card spaced repetition; Math Academy publicly describes a curriculum knowledge graph that estimates a student’s “knowledge frontier” and tracks per-topic spaced-repetition stability. Of those three, Math Academy is the strongest live example of graph-shaped learner modeling, but it is a platform-owned curriculum graph, not a personal PKM vault. Recall and RemNote are useful precedents, but their memory models are mostly question/card performance, not topology-change-based mastery inference. ([Recall Documentation][7])

In the 2024-2026 literature, there **is** movement toward what you want, but it is mostly prototype land. A 2024 UMAP paper builds transparent learner models from personal knowledge graphs in CourseMapper by letting learners mark “Did Not Understand” concepts. A 2025 English-learning system uses a knowledge graph plus graph propagation plus forgetting, and the paper says it was deployed in a WeChat Mini Program. A 2025 paper on KG-based personal learning path planning explicitly combines a graph with learner-mastery estimates. LAK 2025 also accepted a paper on using KGs and LLMs to track learning trajectories. That is real research. It is not yet a settled, widely deployed personal-vault product category. ([University of Duisburg-Essen][8])

On **xAPI / Caliper**: these matter, but not as temporal KGs. Caliper is a standard for collecting learning-activity and product-usage data for dashboards and advising. ADL’s TLA documentation describes xAPI as the standard for capturing learning activity streams via an LRS. So these standards are useful for the evidence layer: they tell you how to log learner events. They do not, by themselves, solve learner-state inference or graph-based forgetting. ([1EdTech][9])

What this means for your repo:

Your current Track B vs Track C tension is real. My read is that **Track B plus a thin temporal extraction layer is the right move now**$_{80%}$.

The reason is simple: the hard part in your personalized-learning/vault use case is **not** storing time on edges. The hard part is defining what counts as evidence that a learner knows, half-knows, forgot, restructured, or mislearned something. None of Graphiti, TigerGraph, Neo4j, or TerminusDB gives you that for free. The live learning systems that get closest use quiz performance, explicit learner signals, diagnostic questions, or repeated review events, not graph topology drift alone. ([Zep Documentation][2])

So the clean architecture for this repo looks like this:

Keep Markdown as the canonical human surface. Build the graph as a derivative layer, not the source of truth. Extract claims/concepts/relations one way from notes, chats, reviews, and teaching interactions. Log learner events explicitly, ideally in an xAPI/Caliper-like shape: `encountered`, `linked`, `explained`, `recalled`, `failed_recall`, `contradicted`, `revised`. Then compute mastery/decay from those events. Do **not** assume graph topology changes are enough. They usually are not. ([ADL][10])

If you want a backend choice right now: use **Graphiti** only if you specifically want automatic time-aware claim extraction from chats/docs and you can tolerate rough edges. Use **Neo4j** if you want full control and are willing to hand-model time. Use **TerminusDB** if revision history, branching, and human-visible diffs matter more than semantic fact validity. Use **TigerGraph** only if you actually have enterprise-scale event throughput or traversal latency needs. For a local personalized-learning vault, TigerGraph is almost certainly too much, and a full persistent KG is probably premature unless your note/event volume gets large enough that inference-time assembly stops being workable. ([GitHub][1])

Blunt version: temporal KGs exist; temporal personal learner models barely do. John’s infrastructure instinct was directionally right, but it overstates what current graph systems buy you. They buy you history. They do not buy you pedagogy.
