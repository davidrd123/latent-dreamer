Using your repo as the guardrail: I’m not redoing discourse graphs, nanopublications, ORKG, etc. You already have that. The open hole is the practical dock. On that question, the answer is pretty clear: the deployed path today is mostly **file-first**, not graph-first. In practice, people are docking Markdown/Obsidian vaults into LLMs by making the vault readable as a local working tree, adding durable instruction files, and then layering thin bridges for search/edit/frontmatter when needed. That fits your Track B much better than a heavyweight KG story.

The strongest current implementation of that pattern is **Claude Code**. Anthropic’s docs say Claude Code carries knowledge across sessions with two memory mechanisms: `CLAUDE.md` files and automatic memory. `CLAUDE.md` can live at user, project, and managed-policy scopes; ancestor files are loaded automatically, subdirectory `CLAUDE.md` files lazy-load when Claude enters that part of the tree, and `@path/to/import` lets one memory file pull in others. Anthropic also exposes `.claude/rules/` for path-scoped instructions, keeps auto memory as plain Markdown under `~/.claude/projects/.../memory/`, loads the first 200 lines of `MEMORY.md` at session start, and lets you inspect/edit memory with `/memory`. That is a real PKM docking mechanism, not just “paste some notes into chat.” It means you can treat a vault as a structured filesystem with scoped norms, schemas, and persistent local memory. It also means your repo’s current idea of “files are primary, semantic matching happens at inference time, provenance lives in Markdown” is not a workaround. It is basically the state of the art for practical use right now. ([Claude][1])

Claude Code also matters because it already speaks **MCP**, and Anthropic’s docs explicitly position MCP as the way Claude connects to external tools and data sources. In other words, if you want your vault to stay file-first but still expose note operations, frontmatter edits, vault search, backlinks, or custom retrieval, Claude Code already has the right transport layer. The practical pattern is: keep the vault on disk, use `CLAUDE.md` and rules to teach the agent how to read it, and add MCP only for operations the model should not fake. That is cleaner than building a parallel persistent knowledge graph too early. ([Claude][2])

**Cursor** has the right nouns, but the practical story is shakier. The official docs/search surface says Cursor supports project rules via `.cursor/rules`, MCP connections, `@Docs` for indexed documentation, and automatic inclusion of relevant workspace context. So yes, Cursor can be part of a PKM dock, especially if your vault or a curated vault subset sits inside the workspace and looks enough like a docs/code repo. But public forum reports are not flattering: users have repeatedly reported that `@Docs` are globally scoped rather than project-scoped, and that doc indexing/retrieval has been unreliable or outright broken for stretches of 2025 into 2026. So Cursor is usable, but I would not treat it as the primary memory substrate for a serious vault workflow. It is better as an editor with contextual assists than as the authoritative docking layer. ([Cursor][3])

**Gemini** is good at a different problem. Google’s docs show three strong mechanisms: URL Context for fetching and analyzing URLs, native document understanding for PDFs and other files, and a Files API plus context caching for large repeated corpora. Gemini can also be grounded with Google Search for live web retrieval. That makes Gemini quite good for “I have a textbook / packet / exported vault slice / PDF bundle, now reason over it,” especially when you want very large context and strong document handling. What it is **not** good at, at least from the public tool surface, is native Obsidian coupling. There is no first-class “Gemini, live inside my vault” equivalent to Claude Code’s local filesystem + `CLAUDE.md` + memory model. The practical Gemini pattern is therefore exported-context and session-grounding: upload files, point at URLs, cache the corpus, maybe wrap it in a Gem or app flow. Useful, yes. A live PKM dock, not really. ([Google AI for Developers][4])

On **Obsidian MCP servers**, the ecosystem is real but fragmented. There is no settled winner or standard package everybody uses. The repos mostly split into two families: servers that talk to Obsidian through the **Local REST API** plugin, and servers that read/write the vault directly from the filesystem. The first family is richer but inherits Obsidian-running/plugin/API-key/certificate hassle. The second family is architecturally cleaner for file-first workflows, but usually thinner. ([GitHub][5])

The most convincing active general-purpose bridge I found is **bitbonsai/mcpvault**. It has roughly **700+ GitHub stars**, commit activity visible in **March 2026**, and explicitly markets itself as a universal AI bridge for Obsidian vaults. It is active enough that I’d call it the best current “alive” option. The maturity caveat is that the repo shows **no published releases**, so it still has a somewhat rolling-head feel rather than “boring infrastructure.” ([GitHub][6])

The repo with the biggest mindshare is **MarkusPfundstein/mcp-obsidian** at about **3k stars**. That sounds great until you check freshness: the visible commit history is old enough that the latest listed commit was **April 2025**, while issues were still active into late 2025 and early 2026 with reports about schema compatibility, macOS build trouble, certificate pain, and timeouts on larger vaults. My read is blunt: high adoption, but maintenance risk. I would not start there unless the ecosystem around it matters more than freshness. ([GitHub][7])

A solid mid-sized option is **cyanheads/obsidian-mcp-server**, around **397 stars**. It exposes a broad CRUD/search/frontmatter/tag tool surface and also rides the Local REST API path. The visible commit history I found was **June 2025**, so it is not dead, but it is not the freshest either. I’d classify it as usable, broad, and somewhat less alive than mcpvault. ([GitHub][5])

The repo that most cleanly matches your repo’s philosophy is **Piotr1215/mcp-obsidian**. It is tiny, about **16 stars**, but the important part is architectural: it explicitly works by **direct filesystem access** and says it does **not** require Obsidian itself to be running. For a file-first vault where Markdown is the source of truth and the editor is optional, that is the right shape. It is less mature socially, but more aligned with your “don’t create a second hidden system unless you have to” instinct. ([GitHub][8])

One repo to skip is **aaronsb/obsidian-semantic-mcp**. It is interesting because it pushed toward semantic/state-aware workflows, but the repo is **archived and read-only** as of **September 2025**. That is exactly the sort of thing people cite in “look, semantic MCP!” conversations and then quietly fail to ship with. Don’t build around archived glue. ([GitHub][9])

So what does “PKM as a structured context source rather than a RAG corpus” actually mean in today’s deployed stack? Mostly this: the structure lives in **directory layout, frontmatter, stable note IDs/titles, instruction files, and tool affordances**. It usually does **not** live in a separate synchronized graph database. Claude Code’s scoped memories, imports, and path rules are one version of that. Obsidian MCP servers exposing note/frontmatter operations are another. Gemini’s file + URL + caching stack is another. MCP is the transport. Markdown plus conventions is the substrate. That is not as glamorous as “dock my mind-graph into the model,” but it is the part that works. ([Claude][1])

On **Peter Danovich**, I could not verify the anecdote. I searched for public traces of “Peter Danovich + Gemini + textbook” and did **not** find a writeup, repo, talk, or doc set I’d trust enough to summarize as fact. The closest public analogs I found were Google’s own “reimagining textbooks with generative AI” material and Gemini-for-Education/Gems positioning, but those are not the Peter Danovich example from your notes. So, for now, that example should be treated as **unconfirmed hearsay**, not as an anchor case study. ([Google Research][10])

My practical recommendation for *this* repo is simple.

First, keep the vault and repo as the source of truth. Do not rush into Graphiti or any parallel KG just to get “docking.” The practical ecosystem is backing your file-first bet, not contradicting it.

Second, if the main agent is Claude Code, lean into `CLAUDE.md`, `.claude/rules/`, imported memory files, and scoped subdirectory memories. That gives you a durable, inspectable docking layer with almost no extra infrastructure. ([Claude][1])

Third, add an Obsidian MCP server **only** if you need live note operations from outside the editor. If you want the safest current general bridge, use **mcpvault**. If you want the most philosophically aligned file-first bridge, test **Piotr1215/mcp-obsidian** first. ([GitHub][6])

Fourth, use Cursor as a secondary surface, not the memory authority. It is fine for editing and workspace-level help, but the docs/indexing story still looks too flaky to trust as your core PKM dock. ([Cursor][11])

Fifth, use Gemini for curated packets: exported Markdown bundles, PDFs, textbooks, reading packs, or public URLs with caching. That is where Gemini is strong. “Live Obsidian brain” is not. ([Google AI for Developers][4])

Net: **usable docking exists now**, but it looks more like **Markdown + scoped memory + thin bridges** than like a settled personal-knowledge operating system. Your Symbiotic Vault direction is closer to what actually ships today than the more ambitious graph-intersection vision.
