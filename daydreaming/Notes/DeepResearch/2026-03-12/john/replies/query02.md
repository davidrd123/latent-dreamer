Here’s the blunt read.

Graphiti is real, but the pitch gets inflated. What it actually is: a temporal context-graph engine for agent memory. It ingests episodes (`text`, `message`, `json`), extracts entities and facts, keeps provenance back to the source episode, stores temporal metadata on edges, optionally builds community summaries, and retrieves with hybrid semantic + keyword + graph-aware search. That is a serious system. It is not, by itself, proof that “knowledge graphs beat vector DBs” or that your PKM should become a persistent graph. ([GitHub][1])

The first correction to John: “Graphiti over Neo4j” is partly stale. Neo4j is still the **primary** backend and the docs position Neo4j Community as suitable for development, testing, and smaller production workloads, but Graphiti now also supports FalkorDB, Amazon Neptune + OpenSearch Serverless, and Kuzu. So Neo4j is no longer a hard dependency. It is the default center of gravity. ([Zep Documentation][2])

What John gets right, wrong, and overstated:

* **“Graphiti is an agent memory/KG layer, not an orchestrator.”** Right. That is exactly how the docs and paper position it. ([GitHub][1])
* **“Temporal evolution is the killer feature.”** Mostly right, but narrower than he says. Graphiti has a real bi-temporal model, tracks fact lifecycles, and invalidates older contradictory edges when new temporally-overlapping facts arrive. That is temporal fact management. It is **not** automatic modeling of learner mastery, knowledge decay, or conceptual change in the richer educational sense. ([arXiv][3])
* **“Entity resolution cleans messy notes into clean concepts.”** Overstated. The paper describes a heuristic pipeline: extract entities, embed names into a 1024-d space, do cosine + full-text candidate search, then use an LLM prompt to decide duplicates. I found no public precision/recall benchmark for Graphiti’s entity resolution specifically. Public issues report duplicate entities and nondeterministic quality concerns, and a March 2026 issue points out that without developer-supplied `entity_types`, all entities collapse to generic `Entity` labels even though edges get richer free-form relation types. So “automatic ontology learning” is only partially true. ([arXiv][3])
* **“Frames become literal subgraphs/communities.”** Partly true in a loose sense, not in the strong sense John implies. Communities exist, but they are generic clusters of connected entities with summaries, not semantically privileged “frames.” Worse, the implementation story is messy: the paper says Graphiti uses **label propagation instead of Leiden**, while the current docs say communities are built with **Leiden** and then incrementally updated with a label-propagation-inspired rule. That inconsistency is a smell. ([arXiv][3])
* **“Community detection is production-stable.”** No. Public issues show `update_communities=True` breaking, and another issue reports `build_communities()` OOM-crashing on graphs with 1000+ entities because it launched one query per entity; the example given is 1,210 entities → 1,210 queries → process killed by the OS. Do not build user-facing meaning on top of community nodes until this area looks calmer. ([GitHub][4])
* **“Docking as graph intersection / shortest path to learning goals.”** I did not find that as a built-in Graphiti capability in the docs, examples, or paper. Treat that as an architecture you would build *on top of* Graphiti, not something Graphiti already gives you.

On scale, the answer is annoying: Graphiti does **not** publish the benchmark you asked for, namely “100 vs 10K vs 100K nodes.” So you cannot honestly quote a clean crossover point from their materials. The hard signals are operational instead. Graphiti keeps default concurrency low (`SEMAPHORE_LIMIT=10`) to avoid LLM 429s; `add_episode_bulk` is faster but explicitly skips edge invalidation; one Feb 2026 issue reports ~60 minutes to bulk-ingest 100 records with a custom ontology; an older issue shows 10 messages taking around 2 minutes; and Zep’s own scale write-up says that under real growth they had to move vector search and BM25 out of the graph DB because the graph DB was doing too much. That is not “Neo4j plus magic.” That is real infra work. ([GitHub][1])

Also, don’t let Zep marketing numbers bleed into Graphiti judgments. The official docs themselves distinguish managed **Zep** from open-source **Graphiti**: Zep is the managed platform with preconfigured retrieval and sub-200ms performance claims; Graphiti is the self-hosted engine whose performance “depends on your setup.” That distinction matters. ([GitHub][1])

Failure modes are not hypothetical. Graphiti warns it works best with structured-output LLM services and says other services may produce incorrect schemas and ingestion failures. GitHub issues show searches returning no results under OpenAI-compatible Qwen + Ollama setups despite edges existing, FalkorDB namespace/search bugs, confusion around mandatory `group_id`s, and requests for a no-LLM ingestion mode because LLM extraction was slow, costly, and reduced data quality for already-structured relational sources. ([GitHub][1])

Public user chatter is mixed, which is what you’d expect from a real but rough system. One Reddit user complained about loose nodes, skipped plain relationships, repeated context, and repeated reattachment of company/name/role facts. Another production user said Zep worked well after early performance issues. Another cited scaling, schema changes, and cloud costs as pain points. GitHub issues are the better signal because they pin complaints to reproducible failure modes. ([Reddit][5])

Direct comparison set:

* **Microsoft GraphRAG** is for building a hierarchical graph over mostly static corpora, then answering with global/local/DRIFT search and community reports. Its indexing pipeline is chunk → extract graph/claims → detect communities → generate reports. Standard GraphRAG uses LLMs for entity/relationship extraction and summarization; FastGraphRAG swaps in noun-phrase/co-occurrence heuristics and is cheaper but noisier. Microsoft estimates graph extraction is about 75% of indexing cost. This is better than Graphiti for large static document collections and “summarize the landscape” questions. It is worse for constantly changing facts and explicit temporal invalidation. ([Microsoft GitHub][6])
* **LlamaIndex PropertyGraphIndex** is a lighter, more modular middle ground. It runs configurable `kg_extractors` per chunk, supports schema-guided or free-form extraction, and uses a `PropertyGraphStore` abstraction that can be in-memory or backed by external graph/vector stores. Good when you want some graph structure without signing up for a full temporal memory engine. I did not find Graphiti-style bi-temporal invalidation/community maintenance in its docs. ([LlamaIndex OSS Documentation][7])
* **LangChain graph stores** are mostly glue, not a coherent memory system. `LLMGraphTransformer` converts docs to graph docs, `Neo4jGraph` wraps the driver, `GraphCypherQAChain` does NL→Cypher, and `graph-retriever` adds metadata-based traversal over vector stores, even with on-the-fly edges. Good if you already know the exact graph behavior you want to implement. Bad if you want Graphiti’s “temporal memory engine” semantics out of the box. ([LangChain Blog][8])
* **Plain inference-time semantic matching over flat markdown files** is still the default I’d pick for a small-to-medium PKM. By construction it removes graph backends, LLM entity-resolution prompts, community rebuilds, and namespace plumbing. You lose explicit validity windows, automatic contradiction invalidation, and reusable graph neighborhoods. That trade is usually good until you have repeated failures that are specifically temporal or multi-hop. This is an architectural inference from the systems above, not a published benchmark. ([GitHub][1])

So where does Graphiti’s overhead pay off?

My rough cut$_{70%}$: **100 nodes: no. 10K nodes: maybe. 100K nodes: only if this is core infrastructure.** More usefully: Graphiti starts to earn its keep when all three of these are true at once: the same entities recur across many episodes, facts about them change over time, and you repeatedly ask entity-centric or multi-hop questions where getting chronology wrong hurts. If your vault is mostly static notes/documents and the model can just read them at inference time, Graphiti is probably overhead. If your system is closer to “live memory layer over evolving people/projects/business data,” Graphiti becomes more defensible. The evidence here is indirect because Graphiti does not publish the size benchmark you want, but the community OOM issue at ~1k entities, the ingestion-cost reports, Zep’s scale redesign, and an independent 2026 study where mem0 beat Graphiti on efficiency without a statistically significant accuracy gap all point the same way: graph overhead is real and must be justified by workload, not vibes. ([GitHub][9])

Bottom line: if the job is “help an LLM read my Obsidian vault,” Graphiti is usually too much. If the job is “maintain a changing, provenance-linked, entity-centric memory with explicit temporal invalidation and repeatable graph queries,” Graphiti is one of the more serious open-source options, but it is still rough enough that I would not introduce it until flat files or a lighter property-graph layer have already failed in concrete, repeatable ways. ([GitHub][1])

Relax your shoulders.
