I dug further. My view got sharper, not different: the biggest win is to make the agent participate in Clojure’s actual workflow, not just autocomplete text. Official Clojure docs center editor-integrated evaluation and structural editing, and the strongest current Clojure-specific agent tools are basically bridges to those two capabilities. I did not find controlled studies specific to Clojure coding agents, so the evidence here is mostly official docs, current tool docs, one useful n=1 practitioner report from a 250k-300k LOC Clojure codebase, and general long-context research. ([Clojure][1])

## What seems to matter most

1. **Give the agent a REPL and structure-aware editing.**

Official Clojure guidance says the normal loop is: open a file, connect a REPL, evaluate in-editor, repeat, and it explicitly recommends structural editing because Clojure code is a tree of balanced forms, not plain text. If your agent is editing by raw substring replacement with no REPL, it is fighting the language. ([Clojure][1])

For CLI assistants, the current best-looking default is **ClojureMCP Light** first, then full **ClojureMCP** with `:cli-assist` if you need structural editing fallback. Its docs say Light gives REPL integration plus delimiter repair while preserving native inline diffs, and full ClojureMCP adds structure-aware edits using parinfer, cljfmt, and clj-rewrite. In VS Code/Calva land, **Backseat Driver** exposes REPL evaluation plus top-level structural edit tools. ([GitHub][2])

2. **Teach the agent to inspect before it writes.**

A lot of bad Clojure from agents is not syntax failure, it is invented vars, wrong arities, and misunderstanding of map shape or namespace API. The standard library already gives the REPL tools you want an agent to use first: `doc`, `source`, `apropos`, `find-doc`, and `dir`. That lets it ask the runtime what exists before touching code. ([Clojure][3])

This is not theoretical. In a 2026 report from Shortcut, on a 250k-300k LOC Clojure codebase, Ivan Willig says their results improved when they grounded the model in the REPL before writing code and narrowed verification steps away from slow full-suite runs. Treat that as practitioner evidence, not a benchmark. Sample size is one team, one codebase. ([I Willig][4])

3. **Commit the conventions as executable config, not prose.**

The baseline stack should be committed into the repo: `.clj-kondo/config.edn`, `.lsp/config.edn`, formatter config such as `.cljfmt.edn`, and `bb.edn` tasks. clj-kondo’s own docs say `.clj-kondo/config.edn` is the preferred configuration location, and its README shows it catches arity errors, unresolved vars/namespaces, duplicate requires, alias consistency, shadowed vars, missing map keys, and more. clojure-lsp reads project settings from `.lsp/config.edn`, supports formatting via `.cljfmt.edn`, and its clean/sort features explicitly follow the Clojure Style Guide for `ns` forms. Babashka tasks are designed to play the role of `Makefile`, `Justfile`, or `npm run`. ([GitHub][5])

The agent should have one or two deterministic commands it can always run, for example `bb lint`, `bb test:ns`, `bb ci`. If it has to remember a six-part `clojure -X ...` incantation, it will get it wrong sooner or later. That part is my inference, but it follows directly from how these tools are meant to be used. ([Babashka Book][6])

4. **If the codebase uses macros or DSLs, teach the tooling once.**

This is an underused high-leverage move. clj-kondo supports hooks such as `:analyze-call` and `:macroexpand`, running in SCI, so custom macros can be rewritten into forms the linter understands. It also supports exported configs on the classpath via `clj-kondo.exports/...`. clojure-lsp likewise supports classpath configs via `clojure-lsp.exports/...` and `:classpath-config-paths`. If your org has internal macros, export the linter/LSP knowledge from the library itself. Then humans, editors, and agents all see the same semantics. ([GitHub][7])

Without that, agents keep trying to “fix” code that is only opaque to their tools. In Clojure, that cost compounds fast because so much real code lives behind macros. ([GitHub][8])

5. **Do not leave “idiomatic Clojure” implicit. Write the idioms down.**

One of the better practitioner observations I found is Willig’s note that the model tended to drift into imperative style, for example `doseq` plus `atom`, where `map` or `reduce` would have fit better. That matches the style guide almost line for line: prefer higher-order functions to `loop/recur` for ordinary collection transforms; prefer destructuring over index-based access; prefer threading macros for linear data flow; switch to an options map when positional arity exceeds 3 or 4; prefer functions to macros unless you really need control of evaluation; never `def` inside a function; and do not shadow `clojure.core` names. Namespace hygiene matters too: one namespace per file, avoid single-segment library namespaces, start with a complete `ns` form, prefer `:require` over `:use`, and standardize aliases across the repo. ([I Willig][4])

One subtle rule worth writing down because models often get it backward: the style guide says *do not* wrap a function in a useless anonymous function like `#(even? %)`, but it also says prefer a simple anonymous function over `comp`/`partial`/`complement` when that reads more clearly. Models often overproduce pointfree cleverness because it looks “functional.” That is not the house style Clojure programmers usually want. ([Clojure Style Guide][9])

6. **Make data contracts executable at boundaries.**

For public boundaries, especially maps, stop relying on English. Use `spec` or Malli and include real sample inputs/outputs. The spec guide says spec names are always fully-qualified keywords, `explain-data` returns machine-readable failure info, and `stest/check` can auto-generate tests from function specs using test.check. That is good raw material for both humans and agents. ([Clojure][10])

Malli has a nice shape for agent-heavy workflows. Concrete facts: its schemas are data, it supports function schemas, dev-time instrumentation, and clj-kondo integration, and `malli.dev` can emit clj-kondo type configs for instrumented functions. My claim that this is especially agent-friendly is an inference, but a pretty grounded one: inspectable schemas plus runtime instrumentation reduce guesswork a lot. ([GitHub][11])

7. **Optimize verification for feedback speed, not ritual.**

Clojure’s own docs frame the REPL as an interactive environment where you build and debug by running small experiments. That implies the agent’s inner loop should be cheap: evaluate a form, run a targeted test namespace, lint the changed files, smoke-test the path, then only later run the big suite. If every edit triggers minutes of verification, the agent starts batching too much work before it checks anything, which is exactly when bad Clojure piles up. ([Clojure][12])

For pure code, add a layer of property tests. test.check exists precisely to state invariants over generated inputs, and spec’s generators plug into that world. Example-based tests still matter more most of the time, but property tests are very good for catching the “looks fine on three examples” class of agent bugs. ([Clojure][13])

8. **Put durable rules at the front of context, not halfway through a session.**

This was the most interesting crossover with general LLM research. Willig reports that mid-session “skills” were not sticky over long conversations and moved more of the durable guidance into the system prompt. That lines up with *Lost in the Middle*, which found models often do best when relevant information is near the beginning or end of context and worse when it sits in the middle. So yes, write `AGENTS.md`, but keep it short, operational, and high in the context stack. Do not bury your style rules in turn 27. ([I Willig][4])

## The stack I would actually use today

For **Claude Code / Codex / Gemini CLI** on Clojure, my default stack would be:

1. `clojure-mcp-light` for REPL integration and delimiter repair.
2. Full ClojureMCP with `:config-profile :cli-assist` if the repo needs structure-aware edits and better nREPL tooling.
3. Committed `clj-kondo`, `clojure-lsp`, formatter config, and `bb.edn`.
4. A short `AGENTS.md` that encodes repo idioms and verification commands. ([GitHub][2])

For **VS Code + Calva**, the analogous setup is Backseat Driver, because it exposes REPL evaluation and structural edits directly through Calva. One nice design detail there is result-size limiting, which is worth copying even if you build your own bridge: bound collection length and nesting depth so evaluation results do not blow up the model’s context window. ([GitHub][14])

## A sharper `AGENTS.md` than most repos have

```md
# Clojure agent rules

Workflow
- Connect to nREPL before any non-trivial edit.
- Before calling an unfamiliar var, inspect with doc/source/apropos/dir.
- Prefer structure-aware edits. Do not regex-edit balanced forms when a structural tool exists.
- Keep diffs small. Preserve public APIs unless task says otherwise.
- After each patch, run fast checks first: bb lint:file + bb test:ns + a REPL smoke check.

Idioms
- Prefer map/filter/reduce/update/update-in into over atom + doseq for pure transforms.
- Prefer destructuring over nth/first/second ladders.
- Use -> / ->> for linear data flow.
- Switch to opts maps after 3-4 positional params.
- Prefer functions to macros.
- Never def inside a function.
- Do not shadow clojure.core names.
- One namespace per file. Prefer :require with standard aliases.

Data contracts
- Public boundary fns should have spec or Malli schema plus one concrete example input/output.
- Use qualified keywords for public map keys and specs.

Verification
- Do not run the full suite after every edit unless the task asks for it.
- Use targeted checks while iterating, full CI at the end.
```

If I had to compress all this into one sentence: **good Clojure from agents comes from REPL access, structure-aware edits, executable conventions, and explicit data contracts; prompting alone is a smaller lever.**
